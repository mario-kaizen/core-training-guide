What is Core?

Hapana Core (core.hapana.com) is the member management platform used by STRONG Pilates. It handles client records, session scheduling, package and membership management, payments, and reporting across all 142 STRONG locations. After logging in, Core presents an Account Picker — select a location to load that studio's dashboard.

Read-only principle: We never edit, create, or update contacts directly in Core. Core is the source of truth; changes flow one-way into Grow via the Core-to-Grow sync.

Client IDs are base64-encoded. The numeric ID (e.g., 3076262) is encoded as MzA3NjI2Mg== in URLs. You can decode with echo 'MzA3NjI2Mg==' | base64 -d.

Finding a Contact

The Clients page is at /index.php?route=dashboard/clients. There are two search methods and two critical filter traps.

Quick Search Bar

The full-width input at the top of the client table (#clientsearch) searches by Name, Email, Barcode, or Phone. It has a 1500ms debounce — after you stop typing, it fires an AJAX reload of the table. The value gets copied to hidden field #clientnamesearch, then #updatelist (the hidden Filter button) is triggered.

Advanced Filter Bar

Below the search bar, columns can be filtered via a dropdown selector (parent_selection[]) with options: First Name, Last Name, Gender, Birth Month, Age, Email, Phone, Client Status. Match types: contains, begins with, ends with, is.

The Alumni Status Gotcha

The status filter (#filter_status) defaults to Active Clients. A contact who cancelled their membership is an Alumni, not Active. You will not find them unless you switch the filter to Alumni Clients.

Status filter options: Active Clients, Alumni Clients, Removed Clients, Deleted Clients.

Corporate Client LookUp

The Corporate Client LookUp button (#corpClientLookup_btn) toggles to a second DataTable (#clientLookupTable) that searches across all locations in the account. Use this when you know the person exists in Core but don't know which location they're at. Click "Back" to return to the single-location view.

Direct URL (if you have the client ID)

If you already have the base64-encoded client ID, go directly:

/index.php?route=dashboard/clients/assessment&cid={BASE64_CLIENT_ID}

Via API (read-only): GET /v2/customer/client?email={email} or ?phone={phone} or ?barcode={barcode}.

Sessions vs Packages — The "Show Visits" Trick

The most common confusion in Core: the Past Sessions view does NOT show which package paid for each session. It only shows "Paid by credits" without identifying the source package.

The answer is on the Packages section, not the Sessions section.

Every package (active or archived) has two links:

  • "Show Visits" (pkgcreditsbookinghistory) — Shows every session consumed from that specific package with columns: S.N, Booking Date Time, Description (class name + date/time), Location Of Use (Home/Away), Credits Used/Refunded, Status (Attended/Cancelled/Late Cancel/No Show)
  • "View History" (pkgpaymentdetails) — Shows the payment/billing history for that specific package

How to Trace Sessions for a Package

  1. Go to the client detail page
  2. Look at the Active Packages section — or click Archived Packages to see past/expired packages (intro offers, expired class packs, past memberships)
  3. Each package shows: Package Name, Credit Limit, valid for {type} valid until {date}, and credit counts (Available/Booked/Used)
  4. Click "Show Visits" on the target package — this is the definitive list of sessions consumed under that package

For intro offers specifically: The intro offer package will be in Archived Packages if it's expired. Click "Archived Packages" to find it.

For free classes: Look for $0.00 packages (e.g., "Friend Free STRONG Class") in either active or archived. "Show Visits" works the same way.

The session view dropdown (#client-schedules) has 4 options: Booked Sessions, Past Sessions, Cancelled Sessions, Schedule Activity Stream. All require a date range (#schedule_date_from / #schedule_date_to) and clicking the #filer_client_schedule button (note: "filer", not "filter" — this is misspelled in the source). Default range is ~60 days back.

Reports That Look Similar But Need Different Filters

Three groups of near-identical reports that differ by what transactions or statuses they include. Getting these wrong means missing data.

Group A: Revenue Transaction Reports

Three reports, same columns, different scope:

Report Filter Name What It Shows What It Excludes
Net Revenue Detail getNetRevenueDetail Completed/settled transactions only In-processing and failed transactions
Net Revenue + In Processing getAllNetRevenueDetail2 Completed + in-processing transactions Failed transactions only
All Invoices Report getAllTransactionsReport Every invoice created in date range Nothing — shows current status of each

All three share the same columns: S.No, Full Name, Email, Payment Method, Invoice Date, Transaction Category, Description, Transaction Id, Processed By, Sale Referred By, Payment Date, Payment Status, Admin Fee, Failed Payment Fee, Transaction Fee, Tax, Discount, Gross Revenue, Revenue Type, Net Revenue.

Payment Status values: Revenue, Refund, In Processing, Failed, Partially Refunded.

Revenue Type values: Session Package, Membership, No Show Fee, Late Cancel Fee, POS, Gift Card.

When to use which: Use "Net Revenue Detail" for clean revenue numbers. Use "All Invoices" when investigating a specific transaction's current status (a refunded transaction may show as "Revenue" in Net Revenue Detail but "Refund" in All Invoices if refunded after the reporting date). Use "Net Revenue + In Processing" when you need to include pending payments.

Group B: Membership Status Reports

Report Filter Name Default View What You Miss Without Filter Changes
Membership Details getMembershipDetails Active only Pending, Suspended, Cancelled, Complete, Scheduled, Partially Refunded
Client on Memberships getactivemembershippkgs Active only Same — cancelled/suspended members are invisible
Client Not On Membership getnonactivemembershippkgs Not on active membership Clients on suspended/pending memberships appear here even though they technically have one
Cancellation Report cancelledMemberships Cancelled only Active/suspended/pending members

Membership Details is the most complete report. It has a Membership Status filter with checkboxes: Active, Pending, Suspended, Cancelled, Complete, Scheduled, Partially Refunded. You must check all statuses to get the full picture. The default "Active" view misses anyone who cancelled, suspended, or completed their membership.

Membership Details columns: S.No, Full Name, Email, Phone, Address, Postal Code, Member Created Date, Package Name, Package Type, Package Category, Package Price, Package Status, Date Sold, Intro Offer Purchase Date, Auto Renew, Auto Renew Date, Expiry Date, Completed Date, Last Payment Date, Last Payment Amount, Next Payment Date, Next Payment Amount, Member Inactive Date, Cancel Date, Cancellation Reason.

Group C: Payment Failure Reports

Report Filter Name What It Shows
Failed Payments paymentfailed Failed payments still pending (unresolved)
Failed Payment History paymentfailedhistory All failed payments with their current status (may now be resolved)

Same concept as Group A: one shows the current unresolved problem, the other shows the full history including resolved failures.
